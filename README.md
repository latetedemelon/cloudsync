# cloudsync

Folder-routed sync and versioned backup from a Linux host to any storage
backend that rclone supports — S3-compatible object stores (AWS S3,
Backblaze B2, Cloudflare R2, MinIO, Wasabi, …), consumer cloud providers
(Google Drive, Dropbox, OneDrive, pCloud, Box, iCloud, …), protocols
(SFTP/SSH, FTP, WebDAV, SMB, HTTP), or local/union/crypt wrappers. See
https://rclone.org/overview/ for the full catalog of ~70 backends.

`cloudsync` itself is storage-neutral: it talks only to `rclone` and
`restic`. Any remote `rclone config` can create, `cloudsync` can drive.

## What it does

For each folder you define in `mappings.yaml`, `cloudsync`:

- Picks the right **tool**:
  - `mode: sync`  — `rclone sync` (fast mirror, deletions propagate, no history)
  - `mode: backup` — `restic` on top of an rclone remote (snapshots,
    deduplication, client-side encryption, point-in-time restore, retention)
- Picks the right **trigger**:
  - `trigger: scheduled` — systemd timer (any `OnCalendar` expression)
  - `trigger: realtime`  — systemd path unit with a debounced runner
  - `trigger: both`      — realtime for responsiveness, scheduled as a safety net
- Logs to the journal, respects per-mapping retention, and coalesces bursts.

## Architecture

```
              ┌──────────────────────┐
              │ /etc/cloudsync/      │
              │   mappings.yaml      │  ◄── single source of truth
              │   rclone.conf        │
              │   secrets/*.pass     │
              └──────────┬───────────┘
                         │  reads
                         ▼
       ┌───────────────────────────────────────┐
       │ /usr/local/bin/cloudsync              │
       │   run <id>  |  run-all  |  check      │
       │   setup-systemd  → generates units    │
       └──────────┬───────────────┬────────────┘
                  │               │
       dispatches by mode:        │
                  │               │
        ┌─────────▼──────┐  ┌─────▼────────┐
        │ rclone sync    │  │ restic       │
        │ (mirror)       │  │ via rclone:  │
        └─────────┬──────┘  └─────┬────────┘
                  └──────┬────────┘
                         ▼
              any rclone remote  (S3, B2, GDrive,
              Dropbox, OneDrive, SFTP, WebDAV, …)

 Triggers (generated by `cloudsync setup-systemd`):

   scheduled:  cloudsync-<id>.timer  →  cloudsync-<id>.service
   realtime :  cloudsync-realtime-<id>.path
                   → cloudsync-realtime-<id>.service
                       → cloudsync-realtime-tick <id>
                           (flock + debounce + drain loop)
                           → cloudsync run <id>
```

The debouncer (`cloudsync-realtime-tick`) uses a flag file plus `flock` so
that: a burst of filesystem events coalesces into one sync, and any changes
that arrive *during* a sync trigger exactly one follow-up sync afterwards —
no lost updates, no thrashing.

## Setup from scratch

### 1. Install prerequisites

Required on the host:

- Linux with systemd
- `python3` with `PyYAML`
- `rclone` — installs the storage backends
- `restic` — only if any mapping uses `mode: backup`

On Debian/Ubuntu:

```bash
sudo apt install python3-yaml rclone restic
```

The distro `rclone` is often a few releases behind. For the current
version, follow https://rclone.org/install/ (a one-line installer script
is available there). Same goes for `restic` (https://restic.net/).

### 2. Install cloudsync

```bash
git clone https://github.com/latetedemelon/cloudsync.git
cd cloudsync
sudo ./install.sh
```

This places the two binaries in `/usr/local/bin`, creates `/etc/cloudsync/`
(with `secrets/` at mode 0700), and drops in a template `mappings.yaml`.

### 3. Configure your rclone remotes

A *remote* is rclone's name for a configured storage destination.
`cloudsync` references remotes by the name you give them here; they can
be of any type — S3, B2, Google Drive, Dropbox, OneDrive, pCloud, SFTP,
WebDAV, SMB, local, or any of the other backends listed at
https://rclone.org/overview/.

Run `rclone config` against the cloudsync config path so the credentials
live alongside your mappings:

```bash
sudo rclone config --config /etc/cloudsync/rclone.conf
```

The interactive menu asks for:

1. **Action** — choose `n` (new remote).
2. **Name** — whatever you want to call it (e.g. `primary`, `offsite`,
   `photos-b2`). This name is what you'll put after `destination:` in
   `mappings.yaml`.
3. **Storage type** — pick from the numbered list of ~70 backends.
4. **Backend-specific options** — usually one of:
   - *OAuth cloud* (Google Drive, Dropbox, OneDrive, pCloud, Box, …):
     rclone opens a browser window; approve access and the token is
     saved.
   - *Access-key cloud* (S3, B2, R2, Wasabi, MinIO, …): paste access
     key + secret, pick a region/endpoint.
   - *Server protocols* (SFTP, WebDAV, SMB, FTP): hostname, username,
     and either a password or a key path.

Backend-specific setup details: https://rclone.org/docs/#configure — pick
your backend from the sidebar for a step-by-step.

Repeat `n` for every remote you want to use. When done, press `q` to
quit.

### 4. Verify each remote works

```bash
sudo rclone --config /etc/cloudsync/rclone.conf listremotes
sudo rclone --config /etc/cloudsync/rclone.conf lsd <remote-name>:
```

`listremotes` prints the names you configured. `lsd` lists the top-level
directories at the root of a remote — if this succeeds, rclone can
authenticate and reach the backend. If a remote fails here, fix it
before creating mappings.

For object stores with a flat namespace (S3, B2, etc.), create a bucket
via the provider UI/CLI first, then verify with `rclone lsd
<remote>:<bucket>`.

### 5. Create mappings

Edit `/etc/cloudsync/mappings.yaml`. The shipped template has one
mapping per pattern — sync-with-safety-net, scheduled backup, realtime
sync, weekly backup, fan-out to multiple destinations — change the
remote names to match your `rclone.conf` and adjust paths to suit.

Destination syntax is rclone's: `<remote-name>:<path>`. For `mode:
backup`, restic sits on top (`restic -r rclone:<remote-name>:<path>`)
automatically.

For each `mode: backup` mapping, create a password file:

```bash
openssl rand -base64 32 | sudo tee /etc/cloudsync/secrets/<id>.pass
sudo chmod 0400 /etc/cloudsync/secrets/<id>.pass
```

**Keep these password files safe** — losing one means the corresponding
restic repository is unrecoverable. Back them up to a password manager
or print + seal.

### 6. Validate

```bash
sudo cloudsync check
sudo cloudsync list
```

`check` verifies required tools are on `PATH`, source directories
exist, mapping ids are well-formed, `OnCalendar` schedules are valid
(via `systemd-analyze`), and password files are mode 0400/0600.

## Deploy systemd units

```bash
sudo cloudsync setup-systemd --dry-run   # preview unit files
sudo cloudsync setup-systemd             # write + reload + enable
```

This writes per-mapping units directly (not templated), so each mapping
gets its own clearly-named service/timer/path pair you can introspect:

```bash
systemctl list-timers 'cloudsync-*'
systemctl status cloudsync-documents.timer
systemctl status cloudsync-realtime-code.path
journalctl -u 'cloudsync-*' -f
```

## Operations

Run a single mapping ad-hoc:

```bash
sudo cloudsync run documents
sudo cloudsync run photos
```

Run everything (useful from a manual cron or for catch-up):

```bash
sudo cloudsync run-all
```

List restic snapshots for a backup mapping (substitute the `destination:`
and `password_file:` values from your mapping):

```bash
RCLONE_CONFIG=/etc/cloudsync/rclone.conf \
RESTIC_PASSWORD_FILE=/etc/cloudsync/secrets/<id>.pass \
  restic -r rclone:<remote>:<path> snapshots
```

Restore from restic:

```bash
RCLONE_CONFIG=/etc/cloudsync/rclone.conf \
RESTIC_PASSWORD_FILE=/etc/cloudsync/secrets/<id>.pass \
  restic -r rclone:<remote>:<path> restore latest --target /tmp/restore
```

## Adding, changing, or removing mappings

1. Edit `/etc/cloudsync/mappings.yaml`
2. `sudo cloudsync check`
3. `sudo cloudsync setup-systemd` (re-generates all units idempotently)

If you **remove** a mapping, re-run `setup-systemd --prune` to disable and
delete the orphan units:

```bash
sudo cloudsync setup-systemd --dry-run --prune   # preview what would go
sudo cloudsync setup-systemd --prune             # apply
```

Without `--prune`, orphan units are only reported, not removed.

## Optional: failure notifications

Every generated unit has `OnFailure=cloudsync-failure@%n.service`.
`setup-systemd` drops in a no-op `cloudsync-failure@.service` template on
first run. Override it once to plug in the notifier of your choice (ntfy,
healthchecks, email, Slack, etc.):

```bash
sudo systemctl edit cloudsync-failure@.service
```

```ini
[Service]
ExecStart=
ExecStart=/usr/bin/curl -fsS -m 10 --retry 3 https://hc-ping.com/<uuid>/fail
```

The `%i` instance name is the failing unit (e.g. `cloudsync-photos.service`),
available as `${MONITOR_UNIT}` in the exec environment.

For success pings, use `ExecStartPost=` directly on the per-mapping unit via
`systemctl edit cloudsync-photos.service`.

## Gotchas

- **rclone sync is destructive by design** — test with `rclone check` or
  temporarily set `rclone_flags: ["--dry-run"]` on a new mapping first.
- **Realtime watching is NOT recursive.** systemd `.path` units watch the
  top-level directory only; modifications deep inside subfolders will NOT
  trigger a realtime sync. For deep trees, set `trigger: both` so the
  scheduled safety net catches subfolder changes, or add explicit
  `PathModified=` lines with `systemctl edit cloudsync-realtime-<id>.path`
  for the subdirs you care about.
- **Many consumer clouds throttle aggressively** (OneDrive, Google Drive,
  Dropbox, pCloud). For large initial uploads, add `--tpslimit 10` and
  `--transfers 2` to `rclone_flags` for that mapping. Object stores
  (S3, B2, R2) generally don't need this. Check rclone's per-backend
  docs for recommended flags.
- **SFTP/FTP needs clock sync** — keep NTP enabled on both ends; large
  clock drift breaks rclone's modification-time comparisons.
- **Object stores charge for egress and listings.** Audit your
  mapping's read pattern (especially `--fast-list` behavior on S3/B2)
  against your provider's pricing before running on a large repo.
- **Restic pruning is expensive.** By default `cloudsync` runs
  `restic forget --prune` after every backup. For frequent backups, set
  `retention.prune: never` on the mapping and schedule a separate weekly
  `restic -r <repo> prune` job.
- **Per-mapping run lock.** `cloudsync run <id>` holds
  `/run/cloudsync/<id>.lock` for the duration of the sync. If a scheduled
  run fires while a realtime-triggered run is in progress (or vice versa),
  the second run exits cleanly rather than racing.
