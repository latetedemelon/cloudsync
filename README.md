# cloudsync

Folder-routed sync and versioned backup from a Linux host to multiple cloud
destinations (pCloud, OneDrive, Crunchbits VPS over SFTP — or any other
rclone-supported backend).

## What it does

For each folder you define in `mappings.yaml`, `cloudsync`:

- Picks the right **tool**:
  - `mode: sync`  — `rclone sync` (fast mirror, deletions propagate, no history)
  - `mode: backup` — `restic` on top of an rclone remote (snapshots,
    deduplication, client-side encryption, point-in-time restore, retention)
- Picks the right **trigger**:
  - `trigger: scheduled` — systemd timer (any `OnCalendar` expression)
  - `trigger: realtime`  — systemd path unit with a debounced runner
  - `trigger: both`      — realtime for responsiveness, scheduled as a safety net
- Logs to the journal, respects per-mapping retention, and coalesces bursts.

## Architecture

```
              ┌──────────────────────┐
              │ /etc/cloudsync/      │
              │   mappings.yaml      │  ◄── single source of truth
              │   rclone.conf        │
              │   secrets/*.pass     │
              └──────────┬───────────┘
                         │  reads
                         ▼
       ┌───────────────────────────────────────┐
       │ /usr/local/bin/cloudsync              │
       │   run <id>  |  run-all  |  check      │
       │   setup-systemd  → generates units    │
       └──────────┬───────────────┬────────────┘
                  │               │
       dispatches by mode:        │
                  │               │
        ┌─────────▼──────┐  ┌─────▼────────┐
        │ rclone sync    │  │ restic       │
        │ (mirror)       │  │ via rclone:  │
        └─────────┬──────┘  └─────┬────────┘
                  └──────┬────────┘
                         ▼
                 pcloud: / onedrive: / vps:

 Triggers (generated by `cloudsync setup-systemd`):

   scheduled:  cloudsync-<id>.timer  →  cloudsync-<id>.service
   realtime :  cloudsync-realtime-<id>.path
                   → cloudsync-realtime-<id>.service
                       → cloudsync-realtime-tick <id>
                           (flock + debounce + drain loop)
                           → cloudsync run <id>
```

The debouncer (`cloudsync-realtime-tick`) uses a flag file plus `flock` so
that: a burst of filesystem events coalesces into one sync, and any changes
that arrive *during* a sync trigger exactly one follow-up sync afterwards —
no lost updates, no thrashing.

## Requirements

- Linux with systemd
- `rclone` (required)
- `restic` (only if you use `mode: backup`)
- `python3` with `pyyaml` (`apt install python3-yaml`)

## Install

```bash
sudo ./install.sh
```

Installs the two binaries to `/usr/local/bin`, creates `/etc/cloudsync/`
(with `secrets/` at mode 0700), and drops in the example `mappings.yaml`.

## Configure rclone remotes

```bash
sudo rclone config --config /etc/cloudsync/rclone.conf
```

Create three remotes (pick any names — just match them in `mappings.yaml`):

- **pCloud** — choose `pcloud`, walk through the OAuth flow
- **OneDrive** — choose `onedrive`, OAuth
- **Crunchbits VPS** — choose `sftp`, point at the VPS with key-based auth

Sanity check:
```bash
rclone --config /etc/cloudsync/rclone.conf lsd pcloud:
rclone --config /etc/cloudsync/rclone.conf lsd onedrive:
rclone --config /etc/cloudsync/rclone.conf lsd vps:/
```

## Configure mappings

Edit `/etc/cloudsync/mappings.yaml`. The example file has one mapping for
each pattern (sync-to-pcloud, restic-to-onedrive, realtime-to-vps, a
scheduled-only pair fanning out to two destinations, etc.) — use them as
templates.

For each `mode: backup` mapping, create a password file:

```bash
openssl rand -base64 32 | sudo tee /etc/cloudsync/secrets/photos.pass
sudo chmod 0400 /etc/cloudsync/secrets/photos.pass
```

**Keep these password files safe** — losing one means the corresponding
restic repository is unrecoverable. Back them up to a password manager or
print + seal.

Validate before deploying:

```bash
sudo cloudsync check
sudo cloudsync list
```

## Deploy systemd units

```bash
sudo cloudsync setup-systemd --dry-run   # preview unit files
sudo cloudsync setup-systemd             # write + reload + enable
```

This writes per-mapping units directly (not templated), so each mapping
gets its own clearly-named service/timer/path pair you can introspect:

```bash
systemctl list-timers 'cloudsync-*'
systemctl status cloudsync-documents.timer
systemctl status cloudsync-realtime-code.path
journalctl -u 'cloudsync-*' -f
```

## Operations

Run a single mapping ad-hoc:

```bash
sudo cloudsync run documents
sudo cloudsync run photos
```

Run everything (useful from a manual cron or for catch-up):

```bash
sudo cloudsync run-all
```

List restic snapshots for a backup mapping:

```bash
RCLONE_CONFIG=/etc/cloudsync/rclone.conf \
RESTIC_PASSWORD_FILE=/etc/cloudsync/secrets/photos.pass \
  restic -r rclone:onedrive:restic/photos snapshots
```

Restore from restic:

```bash
RCLONE_CONFIG=/etc/cloudsync/rclone.conf \
RESTIC_PASSWORD_FILE=/etc/cloudsync/secrets/photos.pass \
  restic -r rclone:onedrive:restic/photos restore latest --target /tmp/restore
```

## Adding, changing, or removing mappings

1. Edit `/etc/cloudsync/mappings.yaml`
2. `sudo cloudsync check`
3. `sudo cloudsync setup-systemd` (re-generates all units idempotently)

If you **remove** a mapping, the orphan units stay on disk until you clean
them up manually:

```bash
sudo systemctl disable --now cloudsync-<old-id>.timer cloudsync-realtime-<old-id>.path
sudo rm /etc/systemd/system/cloudsync-<old-id>.* /etc/systemd/system/cloudsync-realtime-<old-id>.*
sudo systemctl daemon-reload
```

## Optional: healthcheck pings

To detect silent failures, wrap `ExecStart` in each service with a
healthcheck ping. The simplest approach is to add a drop-in for the
generated units:

```bash
sudo systemctl edit cloudsync-photos.service
```

```ini
[Service]
ExecStartPost=/usr/bin/curl -fsS -m 10 --retry 3 https://hc-ping.com/<uuid>
ExecStopPost=/bin/sh -c 'if [ "$SERVICE_RESULT" != "success" ]; then curl -fsS -m 10 https://hc-ping.com/<uuid>/fail; fi'
```

## Gotchas

- **rclone sync is destructive by design** — test with `rclone check` or
  temporarily set `rclone_flags: ["--dry-run"]` on a new mapping first.
- **Path units watch a single directory non-recursively by default.** For
  deep trees with writes in subfolders, rely on the scheduled safety net
  or expand the `[Path]` section with additional `PathModified=` lines.
- **OneDrive throttles aggressively** — for large initial uploads, add
  `--tpslimit 10` and `--transfers 2` to `rclone_flags` for that mapping.
- **Clock skew on the VPS** matters for SFTP-based rclone syncs; keep NTP
  enabled.
